# Federated Active Causal Discovery Framework (JAX)

This repository contains the foundational simulation backend for a Federated Active Causal Discovery environment, formulated as a Decentralized Partially Observable Markov Decision Process (Dec-POMDP) entirely in JAX. 

The primary goal of this framework is to allow multiple decentralized Reinforcement Learning (RL) agents to interact with, observe, and intervene on a hidden global causal graph to collaboratively discover its true structural equations.

---

## 🏗️ Phase 1: Core Engine & Dec-POMDP API

Phase 1 established the core Structural Causal Model (SCM) simulator and the multi-agent reinforcement learning interface.

### What was implemented:
- **`src/types.py`**: Static dataclasses defining the SCM configuration, functional parameter weights, the global environment state, and the localized agent observations.
- **`src/functional.py`**: Pure JAX mathematical primitives that dictate how variables influence each other. Supports both Linear and Non-linear Additive Noise Models (ANM) with various noise distributions (Gaussian, Gumbel, Uniform).
- **`src/scm.py`**: The sequential data generator (`sample_scm`). It computes the values of variables strictly following the graph's topological order.
- **`src/environment.py`**: The Dec-POMDP interface that initializes the graph (`init_env`), allows agents to take actions and decay budgets (`step_env`), and aggregates local samples to generate empirical covariance matrices while strictly masking out structurally unobserved variables (`get_observations`).

### Why this specific implementation was chosen:
1. **JAX `lax.scan` for Topological Ordering**: The most mathematically intuitive way to generate data from a DAG is a dynamic `while` loop that resolves parents before children. However, JAX cannot `jit` compile dynamic loops with data-dependent shapes. Instead of resorting to slow Python loops or relying strictly on matrix inversion (which would limit the framework entirely to linear models), we implemented a highly optimized `jax.lax.scan` loop over a pre-computed topological order vector. This ensures total compatibility with `jax.jit` and `jax.vmap` while seamlessly supporting highly complex non-linear ANMs.
2. **`chex.dataclass` Over Standard Dictionaries**: RL environments commonly use Python dicts to pass states. However, JAX requires states to be registered as PyTrees to undergo transformations like `vmap`. Using `chex.dataclass` ensures strict structural typing and native PyTree compatibility without the overhead of manual registration.
3. **Strict Structural Masking**: In `functional.py`, non-linear models explicitly mask their inputs using the graph's true adjacency matrix *before* passing them to the Multi-Layer Perceptrons (MLPs). This guarantees that non-parents physically cannot influence a child variable, ensuring absolute faithfulness to the DAG topology.

## 🔬 Phase 2: Interventions & Statistical Alignment Layer

Phase 2 expanded the engine to support physical graph manipulation, synthetic data generation, and federated server-side harmonization.

### What was implemented:
- **Modular Interventions (`src/types.py`, `src/scm.py`)**: The engine now natively supports a batched `InterventionSpec`. Agents can execute `HARD` interventions ($do(X_i = c)$), `SOFT_SHIFT` interventions (shifting a mechanism's output), and `SOFT_SCALE` interventions (scaling the aggregated influence of parents).
- **Synthetic Generators (`src/generators.py`)**: Pure JAX functions that generate Erdős-Rényi (ER) and scale-free Barabási-Albert (BA) DAGs, guaranteeing they are strictly upper-triangular. It also randomly initializes structurally valid parameter weights for the SCMs.
- **Statistical Alignment (`src/alignment.py`)**: Server-side federated aggregation.
  - `stouffer_z_score_fusion`: Fuses disparate edge-level P-values generated by different agents into a unified Global Z-score.
  - `stitch_global_covariance`: Safely averages localized overlapping covariance matrices together, weighting by sample counts and marking unobserved pairings as `NaN`.

### Why this specific implementation was chosen:
1. **Unified `InterventionSpec` inside the `scan` loop**: Rather than branching logic or utilizing conditional JAX control flow (which slows down execution), all three intervention types are calculated simultaneously as masked vectors inside the `scan` loop. If an intervention type is not active for a node, its mathematical effect resolves to a neutral identity operation (e.g., scale resolves to `1.0`, shift resolves to `0.0`). This guarantees blazing fast performance during batched generation.
2. **Stouffer's Method for P-value Fusion**: In decentralized causal discovery, different agents may run conditional independence tests on the same variables but with vastly different amounts of data (sample budgets). Fisher's method combines p-values evenly, but Stouffer's method allows weighting by sample sizes ($\sqrt{N_k}$). This specific implementation ensures that an agent with 10,000 samples heavily overrules an agent with only 10 samples when fusing statistical conflicts.
3. **Sequential BA Graph Generation**: Generating Barabási-Albert graphs dynamically requires preferential attachment (sampling based on current degrees). This is historically hostile to static graph execution. We solved this by using the Gumbel-max trick combined with `jax.lax.top_k` inside a `lax.scan` loop, allowing JAX to build massive scale-free causal graphs natively on a GPU/TPU without falling back to CPU-bound NetworkX functions.

## 🧭 Phase 3: Graph Evaluator & PAG Tracking Engine

Phase 3 introduces the causal reasoning engine and wraps the entire decentralized architecture into a standardized, Multi-Agent Reinforcement Learning (MARL) interface.

### What was implemented:
- **`src/pag.py`**: The Partial Ancestral Graph (PAG) tracking engine. It initializes an unoriented, fully connected graph using a NumPy matrix encoding `{0: NULL, 1: TAIL, 2: ARROW, 3: CIRCLE}`. It actively interprets conditional independence matrices generated under $do(X)$ interventions to orient circle marks ($\circ\!\!-\!\!\circ \to \to$) and propagates constraints using FCI (Fast Causal Inference) Meek orientation rules.
- **`src/rewards.py`**: Computes an information-theoretic reward function. It rewards positive ambiguity resolution ($\Delta \text{Circles}$), minus the joint action cost, minus heavy penalties for structural violations (e.g., creating cyclic dependencies).
- **`src/evaluator_env.py`**: A high-level `FederatedCausalEnv` wrapper designed with PettingZoo and Farama Gym paradigms in mind. It seamlessly bridges the discrete multi-agent RL actions into JAX `InterventionSpec` tensors, steps the JAX simulation backend, triggers federated covariance stitching, computes dynamic pseudo p-values, runs the CPU PAG tracker, and returns perfectly isolated dictionary state spaces.

### Why this specific implementation was chosen:
1. **Hybrid Architecture (JAX + NumPy)**: While JAX is unparalleled at compiling massive batched SCM generation tasks, it is notoriously bad at compiling deeply recursive, dynamic graph traversal algorithms like Depth-First Search (DFS) or full FCI causal constraint propagation. Rather than contort the orientation logic into statically sized JAX padding loops that would stall the accelerator, we deployed a Hybrid Architecture. The `FederatedCausalEnv` steps the heavy data generation on the accelerator via `jax.jit`, pulls the stitched covariance back, and rapidly runs the graph algorithms dynamically in pure Python/NumPy on the CPU.
2. **Explicit 2D Matrix Encoding**: Modern graph libraries (like NetworkX) carry significant overhead and state allocation penalties during deep RL training steps. By manually encoding the PAG marks strictly into an optimized 2D integer matrix, we keep memory profiles tiny and perform edge orientation updates at speeds magnitudes faster than standard graph object manipulation.

## 🧠 Phase 4: MARL Control Layer (QMIX)

Phase 4 completed the full environment by implementing the Deep Multi-Agent Reinforcement Learning framework. We adopted a Centralized Training with Decentralized Execution (CTDE) paradigm using Flax and Optax to train cooperative policies.

### What was implemented:
- **`src/marl/agent.py`**: The independent, decentralized deep Q-networks. They take extremely localized, partially observable snapshots of the graph (including a specialized $-1\text{e}9$ action masking penalty for out-of-jurisdiction or over-budget actions) and output values for intervention targets.
- **`src/marl/mixer.py`**: A complex QMIX Centralized Hypernetwork. During training, it views the true global state and dynamically generates the weights for a mixing network that combines all individual agent Q-values into $Q_{\text{tot}}$.
- **`src/marl/trainer.py` & `src/marl/buffer.py`**: A fully automated Temporal Difference (TD) learning loop with a dynamic episodic replay buffer. The trainer manages double Q-learning target networks and executes `optax.adam` optimization steps over massive trajectory batches.

### Why this specific implementation was chosen:
1. **Strict Monotonicity via Absolute Weights**: For decentralized policies to successfully converge on the optimal joint action, the QMIX algorithm mandates $\frac{\partial Q_{\text{tot}}}{\partial Q_k} \ge 0$. We guaranteed this mathematically by enforcing absolute value activations (`jnp.abs`) on all weights generated by the hypernetwork. 
2. **Action Masking**: We chose to implement a negative-infinity logit mask rather than simply invalidating transitions after the fact. This prevents agents from wasting optimization capacity attempting to learn the boundaries of their variable jurisdictions.

### Running the Environment & Training

**1. Verifying the Architecture**
The architecture is fully verified via automated tests. You can run the full suite using `pytest`:
```bash
python -m pytest tests/ -v
```

**2. Launching the Trainer**
You can initialize the `FederatedCausalEnv` and attach it to the `QMIXTrainer` as shown in `tests/test_marl.py`. The typical workflow is:
1. Initialize the `SCMConfig` and generate a graph via `src.generators`.
2. Wrap the backend in `FederatedCausalEnv`.
3. Initialize the `MLPAgent` and `QMIXMixer`, passing them to the `QMIXTrainer`.
4. Run standard multi-agent rollouts (`env.step()`) to populate the `TrajectoryBuffer`.
5. Run `trainer.train_step()` periodically using sampled minibatches to update the decentralized policies.
